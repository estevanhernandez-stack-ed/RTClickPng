## Checklist

- [ ] **1. Decoder binaries acquired + libheif LGPL compliance verified**
  Spec ref: `spec.md > Dependencies & External Services`, `spec.md > Open Issues > License review for libheif`, `spec.md > Open Issues > Decoder binary acquisition`
  What to build: Acquire x64 Windows DLLs for libwebp, libavif, libheif, libspng, and libjpeg-turbo. Pick one acquisition pipeline (vcpkg in `x64-windows` triplet, prebuilt releases from each project's GitHub, or hand-built from source with a fixed CMake config) and commit the chosen approach — binaries land in `build/native/` or are produced by a bootstrap script (`build/fetch-native.ps1`). Verify all five DLLs were linked against the same CRT (`/MD` dynamic UCRT matching what .NET Native AOT expects). Perform the libheif LGPL-3.0 review: confirm that MSIX distribution with libheif as a replaceable DLL (not statically linked, user can overwrite) satisfies the LGPL "relinking" requirement. Document the compliance reasoning in `docs/licenses.md`. If libheif compliance cannot be confirmed, swap to a permissively-licensed HEIC decoder (e.g., a libavif-HEIC path, or drop HEIC from v1 and note in PRD).
  Acceptance: All five DLLs present in `build/native/` (or reliably fetched by bootstrap script). `dumpbin /imports build/native/*.dll` confirms matching CRT across all binaries. `docs/licenses.md` documents license + compliance path for each library. libheif path is formally accepted or formally swapped — no ambiguity remaining.
  Verify: Run `dumpbin /dependents build/native/libheif.dll` (and each other DLL) — confirm dependency on `ucrtbase.dll` / `vcruntime140.dll` and no mixed CRT surprises. Open `docs/licenses.md` and confirm every library has a License + Compliance Path entry. If libheif was swapped, confirm `spec.md > Open Issues` has the decision logged.

- [ ] **2. Engine project skeleton + CLI contract locked**
  Spec ref: `spec.md > Converter Engine (C# / .NET 10 Native AOT) > Program.cs — CLI dispatch`
  What to build: Create `src/Engine/Engine.csproj` with `PublishAot=true`, `TargetFramework=net10.0-windows10.0.22621.0`, `RuntimeIdentifier=win-x64`. Implement `Program.cs` as a hand-rolled `args[]` switch (no third-party CLI parser) that dispatches to `Convert.cs` and `Copy.cs` handlers. Both handlers are STUBBED — they parse their arguments, write `"TODO"` to stdout, and exit 0. Implement return-code scheme from the spec (0 success, 1 generic, 2 source-not-found, 3 format-unsupported, 4 overwrite-denied, 5 output-failed, 10 uncaught). Create `src/Shared/Shared.csproj` with `SettingsSchema.cs` (schemaVersion = 1, two bool fields) and `Paths.cs` (helpers for resolving `%LOCALAPPDATA%\Packages\<PFN>\LocalState\settings.json`). Commit fixture images in `tests/Engine.Tests/fixtures/`: `sample.webp`, `sample.avif`, `sample.heic`, `sample-animated.gif`, `sample-multipage.tiff`, `sample-with-icc.jpg`, `sample.bmp`, `sample.png`.
  Acceptance: `dotnet publish -c Release -r win-x64 src/Engine/Engine.csproj` produces a standalone AOT binary (<10 MB before native DLLs). Running `RTClickPng.Engine.exe convert foo.webp foo.png` exits 0 with `"TODO"` on stdout. Running `RTClickPng.Engine.exe copy foo.webp` exits 0 with `"TODO"` on stdout. Running with unknown verb exits non-zero with usage info on stderr. `Shared.csproj` builds without errors.
  Verify: Publish the engine, run both CLI verbs against fixture files, confirm exit codes. Measure cold-start time with `Measure-Command { ./RTClickPng.Engine.exe convert test.webp test.png }` — target <100 ms. Confirm binary size reasonable.

- [ ] **3. Engine decoders + encoders + round-trip tests**
  Spec ref: `spec.md > Converter Engine > Decoders`, `spec.md > Converter Engine > Encoders`, `spec.md > Converter Engine > Interop`
  What to build: Implement `IImageDecoder` and `IImageEncoder` interfaces in `src/Engine/Decoders/` and `src/Engine/Encoders/`. P/Invoke bindings in `src/Engine/Interop/`: `LibWebp.cs`, `LibAvif.cs`, `LibHeif.cs`, `LibSpng.cs`, `LibJpegTurbo.cs` — use `[LibraryImport]` with `DllImportSearchPath.AssemblyDirectory` so bundled native DLLs are found. Decoder classes: `WebpDecoder`, `AvifDecoder`, `HeifDecoder`, `TiffDecoder` (pure C# or WIC-backed — pick one per `spec.md > Open Issues`), `BmpDecoder` (pure C#), `GifDecoder` (pure C#, first frame only). Encoder classes: `PngEncoder` (libspng, iCCP chunk support), `JpegEncoder` (libjpeg-turbo, quality 92, APP2 ICC support). xUnit test project `tests/Engine.Tests/Engine.Tests.csproj` with `DecoderTests.cs` and `EncoderTests.cs` — decode each fixture, re-encode to PNG, assert output dimensions match source and a known-pixel sample is within tolerance. Wire up `Convert.cs` and `Copy.cs` to actually use decoders + encoders end-to-end (still stubbed on ICC/EXIF handling — that's item 4).
  Acceptance: All seven decoders produce correct RGBA bitmaps for their respective fixture. PngEncoder and JpegEncoder round-trip a test bitmap to disk without data loss. xUnit tests pass on every fixture. `Engine.exe convert sample.webp sample.png` now produces a real PNG file that opens in Photos / Paint. `Engine.exe copy sample.webp` writes real PNG bytes to stdout with a 4-byte length prefix.
  Verify: `dotnet test tests/Engine.Tests/` green. Convert every fixture by hand: `for each fmt: Engine.exe convert sample.<fmt> out.png; start out.png`. Confirm each opens correctly in Windows Photos. For `copy`: `Engine.exe copy sample.webp > out.bin`, verify first 4 bytes are length prefix, skip past it, and confirm remaining bytes form a valid PNG via `file out.bin` or opening in a hex editor.

  > **CHECKPOINT 1** — After item 3, pause. Summarize: decoders work, AOT binary produces correct output, all fixtures round-trip cleanly. Builder verifies before proceeding to item 4.

- [ ] **4. Engine: ColorProfile + ExifHandler + overwrite policy**
  Spec ref: `spec.md > Converter Engine > ColorProfile`, `spec.md > Converter Engine > ExifHandler`, `spec.md > Key Technical Decisions`
  What to build: `src/Engine/Color/ColorProfile.cs` — reads ICC blob from source decoder, passes to destination encoder. `src/Engine/Metadata/ExifHandler.cs` — parses EXIF tag 0x0112 (orientation) only from JPEG/TIFF/HEIC sources, applies rotate/flip to RGBA bitmap before encode, discards all other EXIF. Wire both into `Convert.cs` and `Copy.cs`. Implement `--overwrite-policy={confirm|force|skip}` flag in `Convert.cs` (default: `confirm` when settings says `confirmBeforeOverwrite=true` and Shell Extension passes the flag through; for Copy, policy doesn't apply). When policy is `confirm` and target exists, engine exits 4 and the shell extension handles the prompt (engine is non-interactive). xUnit tests for: ICC profile preserved through round-trip (decode JPEG with ICC, encode PNG, confirm iCCP chunk present); EXIF orientation 1/3/6/8 correctly applied then stripped; overwrite policy exit codes correct.
  Acceptance: ICC profile from a test fixture JPEG reappears in the iCCP chunk of the output PNG. A rotated-via-EXIF JPEG converts to a PNG that displays the correct orientation (pixel data rotated). Output PNG has zero EXIF (`exiftool` reports none). Overwrite exits 4 when target exists and policy is `skip`; overwrites silently when policy is `force`.
  Verify: Run `exiftool -ICC_Profile:all out.png` on a converted JPEG-with-ICC — confirm profile present. Run `exiftool -Orientation out.png` on a rotated-EXIF source — confirm orientation tag is absent and image renders upright. `dotnet test` green on all new tests.

- [ ] **6. Shell Extension: FileFilter + four IExplorerCommand handlers with full visibility logic**
  Spec ref: `spec.md > Shell Extension > ConvertToPngCommand / CopyAsPngCommand`, `spec.md > Shell Extension > FileFilter`, `spec.md > Shell Extension > SettingsReader`, `prd.md > Epic 3 — Supported input formats`
  What to build: `FileFilter.{h,cpp}` — static lowercase extension allowlist compiled in. `SettingsReader.{h,cpp}` — reads `settings.json` from LocalState folder per invocation, defaults to safe values on missing/corrupt. Each implements `GetState()` returning `ECS_HIDDEN` or `ECS_ENABLED` per the exact visibility rules in `spec.md > Shell Extension > Menu visibility logic`: Convert-PNG hides on `.png` sources; Copy-PNG hides when selection count > 1 and shows on `.png` sources (workflow win); JPEG variants only visible when `settings.showJpegVariants == true`. Add all four CLSIDs to `Package.appxmanifest`. C++ unit tests in `tests/ShellExtension.Tests/` for `FileFilterTests.cpp` covering: each supported extension, unsupported extensions, case variations (`.WebP`, `.Heic`), files without extensions, multi-select edge cases, JPEG-toggle-on/off matrix.
  Acceptance: On a machine with JPEG-toggle OFF (default): right-click on `.webp` → both verbs appear; right-click on `.png` → only "Copy as PNG" appears; right-click on `.txt` → neither appears; multi-select 5 webp files → "Convert to PNG" appears, "Copy as PNG" does NOT.   Verify: Install MSIX on Win11 22H2 VM. Manually test the matrix: `.webp` + `.png` + `.jpg` + `.txt` + `.exe` × single-select + multi-select × JPEG-toggle-on + JPEG-toggle-off — 40 cells, spot-check at least 12 covering the rules above. Run C++ unit tests and confirm all pass.

- [ ] **8. Shell Extension: Copy flow end-to-end (ClipboardWriter + three-format paste verification)**
  Spec ref: `spec.md > Shell Extension > ClipboardWriter`, `spec.md > Key Technical Decisions > Clipboard write via three formats`, `prd.md > Epic 2 — Copy as PNG`
  What to build: `ClipboardWriter.{h,cpp}` owns the three-format clipboard write.   Acceptance: Click "Copy as PNG" on a `.webp` file.   Verify: On the Win11 VM with the eleven target apps installed (or a subset — Teams, Figma, Photoshop, Paint is the minimum quorum), perform the paste test for each.
